# Bundled Resources

Speedwave bundles a set of Claude Code resources that are mounted read-only into the Claude container at `/speedwave/resources/` and injected into `~/.claude/` by `entrypoint.sh` at container start. This document is the single reference for **what** is bundled, **how** it reaches the container, and **how** teams can override it.

## Resource Catalog

| Resource                     | Source path                       | Target in `~/.claude/`       | Mechanism               | Overridable?         | Override method                                             |
| ---------------------------- | --------------------------------- | ---------------------------- | ----------------------- | -------------------- | ----------------------------------------------------------- |
| `output-styles/Speedwave.md` | `claude-resources/output-styles/` | `output-styles/Speedwave.md` | symlink (file)[^1]      | No (read-only mount) | Project-level `.claude/output-styles/`                      |
| `statusline.sh`              | `claude-resources/`               | `statusline.sh`              | symlink[^1]             | No (read-only mount) | Project-level `settings.json` with custom command           |
| `settings.json`              | `claude-resources/`               | `settings.json`              | symlink[^1]             | No (read-only mount) | Project-level `.claude/settings.json`                       |
| `mcp-config.json`            | generated by entrypoint           | `mcp-config.json`            | generate (always)[^2]   | No                   | Regenerated on every container start                        |
| `CLAUDE.md`                  | `claude-resources/`               | `CLAUDE.md`                  | symlink[^1]             | No (read-only mount) | Project-level `.claude/CLAUDE.md` or `/workspace/CLAUDE.md` |
| `skills/`\*                  | `claude-resources/skills/`        | `skills/`                    | symlink (directory)[^1] | No (read-only mount) | Project-level `.claude/skills/` takes precedence            |
| `commands/`\*                | `claude-resources/commands/`      | `commands/`                  | symlink (directory)[^1] | No (read-only mount) | Project-level `.claude/commands/` takes precedence          |
| `agents/`\*                  | `claude-resources/agents/`        | `agents/`                    | symlink (directory)[^1] | No (read-only mount) | Project-level `.claude/agents/` takes precedence            |
| `hooks/`\*                   | `claude-resources/hooks/`         | `hooks/`                     | symlink (directory)[^1] | No (read-only mount) | Project-level `.claude/hooks/` (hooks merge across scopes)  |

**Symlink** means the resource is a symbolic link to the read-only mount at `/speedwave/resources/`. Symlinks are re-created on every container start via `ln -sf` (files) or `ln -sfn` (directories), so they always reflect the latest version shipped with Speedwave. **Generate (always)** means the file is regenerated on every start because it depends on runtime variables (never edit it manually). `mcp-config.json` is the only resource generated inline — all others are symlinked from `claude-resources/`.

\* These directories do not exist in `containers/claude-resources/` yet — the entrypoint logic is ready (`[ -d ... ]` guard) and will activate once they are populated. Symlinks are only created when the source directory exists.

## Filesystem Layout

After `entrypoint.sh` completes, the container filesystem looks like this:

```
/home/speedwave/.claude/              ← user-level (Speedwave-controlled)
├── output-styles/
│   └── Speedwave.md → /speedwave/resources/output-styles/Speedwave.md  ← symlink (file)
├── statusline.sh    → /speedwave/resources/statusline.sh               ← symlink
├── settings.json    → /speedwave/resources/settings.json               ← symlink
├── CLAUDE.md        → /speedwave/resources/CLAUDE.md                   ← symlink
├── mcp-config.json                   ← generated (MCP hub endpoint)
├── skills/    → /speedwave/resources/skills/     ← symlink (when dir exists)
├── commands/  → /speedwave/resources/commands/   ← symlink (when dir exists)
├── agents/    → /speedwave/resources/agents/     ← symlink (when dir exists)
├── hooks/     → /speedwave/resources/hooks/      ← symlink (when dir exists)
└── ide/                              ← read-only mount (IDE Bridge lock files)

/workspace/.claude/                   ← project-level (optional, team-committed via git — Speedwave never creates this)
├── agents/
├── commands/
├── skills/
├── settings.json
└── CLAUDE.md
```

The volume mounts that produce this layout are defined in `compose.template.yml`:

```yaml
volumes:
  - ${CLAUDE_HOME}:/home/speedwave:rw # persistent home
  - ${PROJECT_DIR}:/workspace # project directory
  - ${RESOURCES_DIR}:/speedwave/resources:ro # bundled resources (read-only)
  - ${IDE_LOCK_DIR}:/home/speedwave/.claude/ide:ro # IDE Bridge lock files (read-only)
```

## Scope Hierarchy

Claude Code resolves settings and resources using a scope hierarchy where **project-level takes precedence over user-level**:

| Scope   | Container path             | Source                | Priority |
| ------- | -------------------------- | --------------------- | -------- |
| Project | `/workspace/.claude/`      | Team's git repository | Higher   |
| User    | `/home/speedwave/.claude/` | Speedwave bundle      | Lower    |

This means teams can override any Speedwave resource by committing a same-named file to `.claude/` in their repo. Hooks are an exception — hook arrays merge across scopes (both user-level and project-level hooks fire).

**Note on `settings.json`:** The user-level `~/.claude/settings.json` is a symlink to the read-only mount, so Claude Code cannot persist settings changes there (writes fail with EPERM). Teams that need custom settings (e.g., enabling/disabling permissions, configuring thinking) should commit a `.claude/settings.json` at the project level — project-level settings take precedence over user-level for the same keys.

See [ADR-022](../adr/ADR-022-bundled-claude-resources-and-project-coexistence.md) for the full design rationale and rejected alternatives.

## Addon Resources

When addons are installed (`SPEEDWAVE_ADDONS` env variable), `entrypoint.sh` symlinks individual files from each addon into the user-level directories. Addon directories are mounted into the container at `/speedwave/addons/<addon-name>/`:

```
/speedwave/addons/<addon-name>/        ← container path (host: ~/.speedwave/addons/<addon-name>/claude-resources/)
├── commands/    ← per-file symlinks into ~/.claude/commands/
├── agents/      ← per-file symlinks into ~/.claude/agents/
└── skills/      ← per-file symlinks into ~/.claude/skills/
```

Unlike core resources (which symlink entire directories), addon resources create per-file symlinks. This allows core and addon resources to coexist within the same user-level directory. Addon resources support `commands`, `agents`, and `skills` only — `hooks` are intentionally excluded because hooks execute shell commands, and allowing third-party addons to inject hooks would expand the attack surface without user consent.

See [ADR-015](../adr/ADR-015-addon-system-open-core-model.md) for the addon system design.

## Persistence Across Container Restarts

The Claude container's home directory (`/home/speedwave/`) is backed by a persistent volume (`${CLAUDE_HOME}:rw`). This affects resource behavior on restart:

- **Symlinked resources** (`statusline.sh`, `settings.json`, `CLAUDE.md`, `output-styles/Speedwave.md`, `skills/`, `commands/`, `agents/`, `hooks/`) — re-created on every start via `ln -sf` (files) or `ln -sfn` (directories). They point to the read-only mount, so their content updates automatically when Speedwave ships new versions. Teams override via project-level `.claude/` (ADR-022 scope precedence).
- **Always-regenerated resources** (`mcp-config.json`) — overwritten on every container start, because the MCP hub port may change between runs. This is the only resource generated inline by `entrypoint.sh`.

## Adding a New Bundled Resource

Checklist for contributors:

1. Add the file or directory to `containers/claude-resources/`
2. Add symlink logic in `containers/entrypoint.sh` (follow existing patterns — `ln -sf` for individual files, `ln -sfn` for directories)
3. Add BATS tests in `_tests/entrypoint/` covering the new resource (verify with `[ -L ... ]` and `readlink`)
4. Update the [Resource Catalog](#resource-catalog) table in this document
5. Run `make test-entrypoint` to verify
